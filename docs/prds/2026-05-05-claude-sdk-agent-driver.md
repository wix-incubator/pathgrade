# PRD: Claude Agent SDK driver — real `AskUserQuestion` handshake

**Date:** 2026-05-05

## Problem Statement

Pathgrade is an evaluation harness for agent skills. Many of those skills are built around an interactive convention: the agent uses a structured user-question tool to ask the user to pick between options, and then branches on the answer. For Creator Kit's `ask-user tool` convention this is the central contract — the question and the consumed answer are the unit of behavior an eval needs to verify.

Today, when the skill under test runs against Claude, pathgrade invokes Claude headlessly through the CLI in print mode with permissions skipped. Under that mode, Claude's built-in `AskUserQuestion` tool is permission-denied: the turn ends, the question gets reconstructed as plain assistant text, and the harness has no way to deliver the user's answer back to Claude as a real tool result. As a consequence:

- A fixture author can assert that Claude *attempted* to ask the question, but cannot assert that Claude *consumed* the answer and continued down the correct branch.
- The transcript shows a synthesized text message where the real tool call should be, which misleads scorers that key on tool-call structure (and which actively mislabels skills that follow the convention as if they were ignoring it).
- Pathgrade has had to build a "blocked-prompt queue" workaround that feeds reaction answers back into a subsequent turn as a user reply — a multi-turn approximation of an in-turn handshake.
- Pathgrade's Claude integration is built on scraping NDJSON from a CLI not designed to be driven this way, which makes every new feature expensive (regex on stream output, per-flag parsing, brittle session-id handling).

The user impact is that interactive convention flows can only be partially verified. The tooling impact is that pathgrade carries a layer of synthesis machinery that exists solely to compensate for a missing channel.

## Solution

Replace the Claude integration's CLI-scraping driver with one built on `@anthropic-ai/claude-agent-sdk`. The SDK fires a `canUseTool` callback whenever Claude invokes `AskUserQuestion`. Pathgrade resolves the question through its existing ask-bus and reaction infrastructure and returns the chosen answer as the tool's real result. Claude receives the answer mid-turn and proceeds along the correct branch. The transcript records the literal `AskUserQuestion` tool name with its real input and the actual answer. Reactions can now express, verify, and assert the entire handshake — invocation, options, answer, branch — in a single round-trip rather than a multi-turn approximation.

The SDK is a typed-protocol wrapper around a bundled per-platform `claude` binary. Pathgrade keeps its subprocess-based sandbox model: the SDK's custom-spawn hook lets pathgrade wrap each Claude subprocess with `sandbox-exec` exactly as today. The bundled binary becomes the default for reproducible eval runs; users can override it for cases where they need to grade a specific local install.

The migration deliberately separates "Claude Code behavior" from "ambient user machine state". Pathgrade uses the Claude Code system-prompt preset so evals exercise the same core prompt stack, tool instructions, coding guidelines, response style, and safety behavior users get in Claude Code. At the same time, the default eval environment is hermetic: fixture-staged project settings and skills are loaded, but host-level user skills, hooks, memory, and local settings are not silently imported unless an explicit compatibility option asks for them. This keeps skill evals faithful to Claude Code while preserving CI reproducibility.

The synthesis machinery that the old driver depended on — blocked-prompt queues, denial reconstruction, the runtime-policy text injection that worked around the missing channel — is removed in the same change, because its reason for existing is gone.

## Implementation Decisions

### Architecture

- The Claude driver is replaced in place. The migration lands in a single change with no transport flag, no opt-in period, and no parallel CLI driver retained.
- The new driver implements the same `BaseAgent.createSession` contract as today. All upstream consumers (managed-session wrapper, conversation runner, fixture loaders, scorers, snapshot projectors) are unchanged.
- The driver is composed of four deep modules with focused interfaces, plus a small MCP-config helper. The driver class itself is orchestration over those modules.

### Module decomposition

- **Sandboxed-claude-spawn module** — a pure function that adapts the SDK's spawn options into pathgrade's sandbox model. Given a working directory, argv, environment, and executable path, returns a spawned process whose argv has been wrapped with `sandbox-exec` on macOS (or passed through unchanged on other platforms) using pathgrade's existing sandbox profile and env-filtering policy. Knows nothing about Claude, the SDK, or evals.
- **Ask-user-bridge module** — the `canUseTool` callback expressed as a pure function. Given an ask-bus reference, a turn number, a tool name, and a tool input, returns a `PermissionResult` (the SDK's actual return type per `sdk.d.ts:1719-1731`). The Claude SDK driver emits live ask-user batches, so `createSession` must enforce `AgentSessionOptions.askBus` with `requireAskBusForLiveBatches(options, 'ClaudeSdkAgent')`; direct low-level sessions without a bus fail fast instead of silently denying or hanging. For `AskUserQuestion`: builds an `AskBatch` inline from the input's `questions` (live lifecycle, source `claude`, sourceTool `AskUserQuestion`; this construction lives inside the bridge module — no shared helper is extracted from the deleted `buildAskBatchFromClaudeDenials` since the SDK already gives the bridge the typed `AskUserQuestionInput` shape directly, eliminating the denial-reconstruction step the old helper existed to perform), emits to the bus, awaits resolution, and branches on the resolved answer's intent: an answered batch translates to `{ behavior: 'allow', updatedInput: { ...input, answers } }` where `answers` matches the SDK's documented `AskUserQuestionOutput.answers` shape — `{ [questionText: string]: string }` per `sdk-tools.d.ts:2542`, with multi-select answers joined as a single comma-separated string per the SDK's own field comment ("multi-select answers are comma-separated"; the precise separator format is the SDK's contract, so the bridge follows whatever the bundled binary expects rather than imposing a stricter "comma-space" assumption); a declined batch translates to `{ behavior: 'deny', message: 'User declined to answer' }` per User Story #29. If the bus resolution rejects (for example `AskBusTimeoutError`), the bridge returns `{ behavior: 'deny', message: <error message> }` so the SDK turn terminates cleanly and the conversation runner reports `completionReason: 'error'` from the thrown driver error. For all other tool names: returns `{ behavior: 'allow', updatedInput: input }`. The SDK's `AskUserQuestionInput` schema enforces 1–4 questions and 2–4 options per question, so the projector and bus translation never need to handle out-of-range cases originating from Claude.
- **SDK-message-projector module** — replaces the existing stream-json parser and tool-event extractor. Consumes the SDK's typed message stream, accumulates assistant text, tool-use blocks, init metadata (skills, slash commands, tools, MCP servers, model, version), per-turn usage including cache tokens, and typed errors. Produces an `AgentTurnResult`.
- **MCP-config-loader helper** — reads pathgrade's existing MCP config JSON file and returns the SDK's `mcpServers` options object. Lives next to the existing `writeMcpConfig` provider (`src/providers/mcp-config.ts`). The file-write step `writeMcpConfig` performs is preserved (not redundant) because the Cursor agent driver still consumes the on-disk path via `mcpConfigPath` (`src/agents/cursor.ts:253-263` — copies into the trial workspace and passes `--approve-mcps`). The Claude SDK driver no longer passes `--mcp-config <path>`; instead the loader reads the same JSON file `writeMcpConfig` wrote and returns the parsed `mcpServers` object for `Options.mcpServers`. The shapes line up directly: `writeMcpConfig` already emits `{ mcpServers: { [name]: { command, args } } }`, and the SDK's `Options.mcpServers: Record<string, McpServerConfig>` accepts `McpStdioServerConfig = { type?: 'stdio'; command: string; args?: string[]; env?: Record<string, string> }` (`sdk.d.ts:1005`). The `type: 'stdio'` discriminator is optional, so the existing JSON parses inline with no field renames.
- **Claude-SDK-agent driver** — orchestrates the above. Owns per-session state (session id, turn counter, ask-bus reference). Per turn, configures the SDK options (cwd, model, mcp servers, custom spawn function, custom can-use-tool, resume id when present), runs `query()`, drains the typed message stream through the projector, and returns the resulting `AgentTurnResult`.

### SDK option choices

- `permissionMode` is `"default"`. The driver always installs a `canUseTool` that auto-allows non-`AskUserQuestion` tools and routes `AskUserQuestion` through the ask-bus. This contract is strictly safer than `bypassPermissions` because it preserves the `canUseTool` callback for every tool decision and removes ambiguity about whether the callback fires for clarifying-question cases under bypass mode.
- `systemPrompt` is set to `{ type: "preset", preset: "claude_code" }`. This is required because the SDK's default prompt is intentionally minimal and does not include the full Claude Code coding guidelines, response style, safety instructions, or environment context. Verified shape against `sdk.d.ts:1586-1621` — the SDK accepts this exact object form, plus optional modifiers (`append`, `excludeDynamicSections`).
- `cwd` is set to the workspace path so skills under `<workspace>/.claude/skills/` are auto-discovered.
- `settingSources` is set to `["project"]` so fixture-staged `.claude/skills/`, `.claude/commands/`, `CLAUDE.md`, rules, hooks, and settings under the prepared workspace are loaded. The SDK's omitted/empty default is `[]` — full isolation, no filesystem settings of any kind (`sdk.d.ts:1546-1554` — *"When omitted or empty, no filesystem settings are loaded (SDK isolation mode). Must include `'project'` to load CLAUDE.md files."*). The driver explicitly opts into project sources rather than relying on the omitted default precisely because fixture-staged skill discovery requires it; user and local sources remain off in the hermetic default.
- A CLI-faithful compatibility option may set `settingSources` to `["user", "project", "local"]`. That option must be documented as non-hermetic because user skills, hooks, settings, local files, and memory can affect results.
- The driver sets an isolated Claude config/home environment and disables auto memory unless the compatibility option is enabled. The isolation is implemented by setting `CLAUDE_CONFIG_DIR` (referenced as the subprocess's transcript/config root at `sdk.d.ts:1282, 3399, 3406`) to a per-trial scratch directory under the workspace (e.g. `<workspace>/.pathgrade-claude-config/`), passed via `Options.env` so it reaches the bundled subprocess. This prevents host `~/.claude.json`, user memory, and other ambient state from leaking into default CI runs. **Note:** the auto-memory toggle is the typed `Options.autoMemoryEnabled: false` option (`sdk.d.ts:4740`), not an env var — the previously assumed `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` does not exist in the SDK's documented surface and would silently no-op if used.
- `allowedTools` is not set, leaving the full Claude tool surface available, including `Skill` and `AskUserQuestion`. **Static verification status:** `Skill` is referenced in the SDK runtime (`node_modules/@anthropic-ai/claude-agent-sdk/sdk.mjs`) but does not appear in the static type definitions, so the existence of a `Skill` tool name cannot be confirmed by types alone. The smoke spike at `tests/claude-sdk-smoke.test.ts` already proves that with `allowedTools` unset and `permissionMode: 'default'` the full tool surface is reachable: `AskUserQuestion` reaches `canUseTool`, project-staged skills are discovered, and `Skill` appears in the init message's `tools` array.
- `model` is forwarded from the existing session option.
- `mcpServers` is built from pathgrade's existing MCP config JSON, parsed into the SDK options shape inline. Existing stdio-based mock servers continue working unchanged.
- `pathToClaudeCodeExecutable` is omitted by default (uses the SDK's bundled per-platform binary), and is set only when the user supplies an override. Precedence is `AgentOptions.claudeCodeExecutable` first, then `PATHGRADE_CLAUDE_CODE_EXECUTABLE`, then bundled default. The option is intentionally run-level rather than fixture-file-level: a fixture cannot select different Claude binaries per trial in this migration.
- `spawnClaudeCodeProcess` is set to the sandboxed-claude-spawn module so every Claude subprocess runs under pathgrade's sandbox profile.
- **Env ownership.** The driver populates `Options.env` (`sdk.d.ts:1183-1185`) with the union of (a) auth pass-through from `resolveCredentials()` / `resolveClaude()` (`ANTHROPIC_API_KEY`, `ANTHROPIC_BASE_URL`, plus any other surfaces added under the Authentication section), and (b) `CLAUDE_CONFIG_DIR` set to a per-trial scratch directory. The SDK forwards `Options.env` into the `SpawnOptions.env` object (`sdk.d.ts:4888`) that pathgrade's custom `spawnClaudeCodeProcess` receives. The sandboxed-claude-spawn module then constructs the subprocess environment by intersecting that incoming env with `SAFE_HOST_VARS` (`src/providers/sandbox.ts:29-31` — `PATH`, `SHELL`, `LANG`, `LC_ALL`, `LC_CTYPE`, `TERM`, `USER`, `LOGNAME`) for ambient host vars and layering the SDK-provided values on top. This keeps `Options.env` as the single typed declaration of what the SDK driver intentionally carries, while the spawn module remains the single enforcement point for sandbox-exec env stripping. The driver does not rely on host-process env leaking through the sandbox: any variable the SDK subprocess needs must appear in `Options.env`.
- `resume` is set on every turn after the first, using the session id reported on the previous turn's result message.

### Token and cost telemetry

- `AgentTurnResult.inputTokens` remains the total input-token value pathgrade reports today: uncached input plus cache creation plus cache read. This preserves existing `LLMPort.addTokens()` aggregation, `TrialResult.input_tokens`, and snapshots/tests that already treat cache tokens as billable/reportable input volume. The SDK-message-projector also populates `cacheCreationInputTokens` and `cacheReadInputTokens` as optional breakdown fields for Claude only.
- `AgentTurnResult.costUsd?: number` is added for providers that expose exact turn cost. The Claude SDK projector populates it from `SDKResultSuccess.total_cost_usd` and `SDKResultError.total_cost_usd`. `buildModelAgentResultLogEntry()` writes `cost_usd` when present so per-turn cost is visible in `session_log`.
- Run-level cost is additive and conservative. During conversation execution, Claude `costUsd` values are accumulated onto the agent's shared telemetry object alongside token usage. `TrialResult.conversation_cost_usd` records the accumulated Claude agent-turn cost at evaluation time. `TrialResult.total_cost_usd` is emitted only when all included components have known cost; until judge LLM providers expose cost, `conversation_cost_usd` is the only guaranteed cost field.

### Capabilities and runtime policies

- `claude.interactiveQuestionTransport` is flipped from `"noninteractive"` to `"reliable"` in the agent capabilities table at `src/sdk/types.ts:416`. Codex and Cursor stay `'noninteractive'` (`src/sdk/types.ts:417-418`).
- The non-interactive runtime policy is removed from Claude's plan. The policy infrastructure remains in place for Codex (exec) and Cursor, which are still subject to the synthesis problem this PRD does not address.
- The runtime-policies-applied field on the turn result is preserved as an empty array for Claude.

### Unmatched ask-user failures

- `AskUserUnmatchedSignal` grows `questionTexts: string[]` in addition to `batchId` and `turnNumber`. `createAskUserHandler()` records the text of every question that fell through to the unmatched path, including first-option fallbacks that degrade to error for free-text or secret questions.
- The conversation runner's unmatched failure detail becomes `unmatched ask_user on turn <turnNumber>: <batchId> (<question text>; <question text>)`. Tests should assert the structured signal includes the text array and the user-facing `completionDetail` includes at least the first unmatched question.

### Deletions

Each item below was confirmed present at the cited location in this branch.

- The deprecated `BlockedInteractivePrompt` type (4 src/ files) and the deprecated `blockedPrompts` field on `AgentTurnResult` (9 src/ files) are removed.
- The `'blocked_prompt'` value of `VisibleAssistantMessageSource` is removed (`src/sdk/types.ts`, plus 6 dependent src/ files).
- The pending-blocked-prompt queue and its dispatch branch in the conversation runner are removed (`src/sdk/converse.ts:82, 265-388` — variable `pendingBlockedPromptQueue` plus its consumers).
- The denial-reconstruction helpers, the formatted-blocked-prompt helper, and the visible-message blocked-prompt branch are removed.
- The Claude permission-denial → ask-batch parser helper is removed (`src/agents/claude.ts:145, 303-331` — the `parsed.denials.filter((d) => d.tool_name === 'AskUserQuestion')` block plus the reconstruction helper around it). The post-hoc `buildAskBatchFromClaudeDenials` function (`src/sdk/ask-bus/parsers.ts:24-79`) and its `ClaudePermissionDenialShape` type are removed; the only live-batch call-site is the ask-user-bridge module, which constructs the `AskBatch` inline from the already-typed `AskUserQuestionInput` and so does not need the shared helper. The Codex live builder (`buildAskBatchFromCodexRequestUserInput`) in the same file is unchanged.
- The PATH-search executable resolver with shim avoidance is removed (`src/agents/claude.ts:41` — `function resolveClaudeExecutable`); the SDK's bundled binary at `node_modules/@anthropic-ai/claude-agent-sdk-<platform>/claude` handles binary resolution.
- Any test files dedicated to blocked-prompt-queue behavior are removed.

### Authentication

- Authentication is delegated to the bundled Claude subprocess, but the migration does not treat consumer `claude.ai` login as a product-safe default for SDK-powered use. The user guide must distinguish local first-party eval usage from productized or third-party SDK usage.
- Today, `src/providers/credentials.ts:resolveClaude()` only forwards two surfaces to the subprocess: `ANTHROPIC_API_KEY` (either user-supplied, host-supplied, or read from the macOS Keychain `Claude Code-credentials` entry and surfaced as `ANTHROPIC_API_KEY`) and `ANTHROPIC_BASE_URL` (the proxy override path Wix CI uses via `APP_ANTHROPIC_BASE_URL`). The sandbox `SAFE_HOST_VARS` allow-list (`PATH`, `SHELL`, `LANG`, `LC_ALL`, `LC_CTYPE`, `TERM`, `USER`, `LOGNAME` — `src/providers/sandbox.ts:29-31`) does not include any auth env vars; auth reaches the subprocess via `spec.env` from `resolveCredentials()`, layered on top.
- This migration extends `resolveClaude()` rather than downscoping the user stories. The resolver pass-through allowlist adds SDK-supported explicit auth and cloud-provider env surfaces: `ANTHROPIC_API_KEY`, `ANTHROPIC_AUTH_TOKEN`, `ANTHROPIC_BASE_URL`, `CLAUDE_CODE_OAUTH_TOKEN`, Bedrock env vars with prefixes `AWS_` plus `ANTHROPIC_BEDROCK_BASE_URL`, Vertex env vars with prefixes `GOOGLE_`, `GCLOUD_`, and `CLOUD_ML_` plus `ANTHROPIC_VERTEX_BASE_URL`, and Foundry env vars with prefixes `AZURE_` plus `ANTHROPIC_FOUNDRY_BASE_URL`. The resolver still treats user-provided values in `AgentOptions.env` as highest precedence and never overwrites them with host values. `apiKeyHelper` remains out of scope because it is a code callback rather than an environment surface; users who need helper-based auth should inject concrete env credentials for pathgrade runs.
- Existing subscription/keychain credentials may continue to work for local CLI-faithful runs, but this is documented as an environment-dependent compatibility path rather than the recommended CI or productized SDK path.

### Multi-turn semantics

- Each turn is one `query()` call. The first turn omits `resume`. Subsequent turns pass the prior turn's `session_id` as `resume`.
- The `prompt` argument to `query()` (`sdk.d.ts:2078` — `string | AsyncIterable<SDKUserMessage>`) is always passed in its string form. Streaming-input is explicitly Out of Scope (see Out of Scope), so the driver never constructs an async-iterable prompt; per-turn user input flows through the `resume`-per-turn model rather than a single long-lived `query()` with a streamed iterable.
- Session-id sanitization is removed because the value is no longer interpolated into a shell command.
- The conversation runner's existing turn-counter and reactions-fired tracking is unchanged.

### Tool event extraction

- The existing tool-name mapping (`src/tool-events.ts:80` — `Skill` → `'use_skill'`), the skill-event enrichment (`enrichSkillEvents` at `src/tool-events.ts:166-182`, imported in `src/agents/claude.ts:5`), and the slash-command detection (`src/agents/claude.ts:407-425` — the `firstMessage.match(/^\/([^\s]+)/)` block that prepends a synthetic `use_skill` event when `<name>` is in the init message's skills array) are all preserved. The inputs they consume are the same structured tool-use blocks they consume today, sourced from the SDK's typed `assistant` messages instead of NDJSON.

### Migration boundary

- The migration is one change. The repo never lives in a half-state with both drivers present.
- The SDK is added to the package manifest as a runtime dependency pinned to a major version. The bundled-binary optional dependency that ships per platform follows the SDK's own dependency graph.
- The user guide is updated to call out that `AskUserQuestion` is unavailable inside subagents (per the SDK's documented limitation), to explain the new `pathToClaudeCodeExecutable` override pathway, to document the Claude Code system-prompt preset, and to document hermetic versus CLI-faithful setting-source modes.

## Testing Decisions

A good test exercises external behavior — the contract a consumer of the module relies on — and remains stable across implementation refactors. Tests should not assert internal helper-function shapes or mock-call sequences when the same contract can be expressed against the module's public input-output behavior. Each deep module has a focused interface that lends itself to that style.

### Modules covered by tests

- **Sandboxed-claude-spawn module.** Tests feed fake spawn options (cwd, argv, env, executable), then assert that the resulting spawned process was launched with the expected argv prefix and a filtered env dictionary. Tests inject a fake spawn delegate so no real subprocess runs. Platform-specific behavior is exercised by stubbing the platform check. Prior art: the codex transport's spawn-and-handshake unit tests for shape, and any existing tests around the sandbox provider for env-filter assertions.
- **Ask-user-bridge module.** Tests feed canned tool-name and input pairs against a mock ask-bus that returns canned `AskAnswer` results, and assert that `CanUseToolResult.updatedInput.answers` matches the SDK's documented shape across single-select, multi-select, and free-text variants. A separate set of tests exercises non-`AskUserQuestion` tool names, asserting pass-through with original input. Tests also assert missing-bus fail-fast at session construction and bus-timeout denial behavior. Prior art: the existing ask-bus handler tests that exercise reaction matching and unmatched-fallback behavior.
- **SDK-message-projector module.** Tests feed canned typed-message sequences (a successful turn, each error subtype, an init message with skills and slash commands, a multi-tool-use assistant turn, hook-lifecycle events, rate-limit events) and assert the resulting turn-result shape including assistant message text, tool events, runtime-policies-applied, totalized input tokens, cache-token breakdown fields, cost, error flags, and visible assistant message. Prior art: the existing tool-events tests for canonical-action mapping, and the existing converse projector tests that turn message sequences into typed outputs.
- **MCP-config-loader helper.** Tests read sample MCP config files (one mock-server variant, one passthrough config-file variant) and assert the resulting `mcpServers` options object shape. Prior art: any existing fixture-loading tests in the providers layer.
- **SDK-option builder.** Tests assert that the default query options include the Claude Code system-prompt preset, hermetic `settingSources`, isolated config/home env, auto-memory disablement, custom spawn, and no `allowedTools` allowlist. A separate test asserts that the CLI-faithful compatibility option intentionally loads user, project, and local setting sources.

### Integration test

- One end-to-end test drives the new Claude SDK agent driver against a mocked SDK harness across two turns. Turn 1 emits an `AskUserQuestion` tool call; the test installs a reaction whose answer maps to a specific option label. Turn 2 emits an assistant message that branches based on the chosen answer. The test asserts the full ask_user tool event shape (questions, options, the supplied answer values, the answer source field set to `'reaction'`), the bus snapshot's resolved batch, the visible assistant message on each turn, per-turn token usage, per-turn `costUsd`, and `agent_result.cost_usd` log output. Prior art: the existing codex app-server agent tests, and the existing converse-ask-user tests.

### Tests that remain unchanged

- The conversation runner's ask-user tests — they exercise the contract this migration preserves and should continue passing without modification.
- The reaction loader and bus handler tests — same rationale.
- The tool-events tests for canonical-action mapping and skill enrichment — input shape is unchanged.

### Pre-implementation verifications (already done, no further work required)

The following claims were verified empirically against `@anthropic-ai/claude-agent-sdk@0.2.100` and the current branch state. Each is a citation, not a TODO.

- **SDK option surface.** Every option this PRD names exists with that exact name in `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts`: `spawnClaudeCodeProcess`, `pathToClaudeCodeExecutable`, `mcpServers`, `resume`, `settingSources`, `permissionMode`, `canUseTool`, `allowedTools`, `cwd`, `model`, `env`, `systemPrompt`, `autoMemoryEnabled`.
- **`systemPrompt` preset shape.** `{ type: 'preset', preset: 'claude_code' }` is accepted (`sdk.d.ts:1586-1621`), with optional `append: '...'` and `excludeDynamicSections: true` modifiers if pathgrade later wants to extend the preset rather than replace it.
- **Auto-memory toggle is a typed option, not an env var.** `Options.autoMemoryEnabled?: boolean` exists at `sdk.d.ts:4740`. `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` is **not** a documented SDK env var and would silently no-op; the SDK Option Choices section has been updated accordingly.
- **`AskUserQuestion` input/output shape.** Input is `{ questions: [{ question, header, options: [{ label, description, preview? }, ...], multiSelect }] }` (`sdk-tools.d.ts:566`, 1–4 questions, 2–4 options each). Output is `{ questions: [...], answers: { [questionText: string]: string }, annotations?: { [questionText]: { preview?, notes? } } }` (`sdk-tools.d.ts:2542`). The `answers` shape matches what the bridge module produces. The SDK comment says "multi-select answers are comma-separated" without specifying a separator format; the bridge follows the SDK's contract rather than imposing a stricter "comma-space" assumption.
- **Cache token field names already match.** SDK's `NonNullableUsage` extends `BetaUsage`, which exposes `cache_creation_input_tokens` and `cache_read_input_tokens` (`@anthropic-ai/sdk/resources/beta/messages/messages.d.ts:521-525`). Pathgrade already folds these exact field names into totalized `inputTokens` at `src/agents/claude.ts:34-35, 223-227` and `src/agents/cursor.ts:127-128, 179-182`. The SDK projector keeps that totalized convention while adding separate optional breakdown fields.
- **`SDKResultError` subtype set is wider than expected.** The four documented subtypes — `error_during_execution`, `error_max_turns`, `error_max_budget_usd`, `error_max_structured_output_retries` (`sdk.d.ts:2935`) — are all surfaced as typed-error events by the projector. (User Story #19 has been updated to name all four explicitly.)
- **`AskUserQuestionOutput.annotations` is currently unused.** The SDK exposes `annotations?: { [questionText]: { preview?, notes? } }` for free-text user notes and preview-content tracking. Pathgrade does not consume it today and will not in this migration; the field is available for a future enhancement (visual previews on options) and is explicitly Out of Scope here.
- **MCP config shape parity.** The JSON `writeMcpConfig` already writes (`{ mcpServers: { [name]: { command, args } } }`) parses inline into `Options.mcpServers: Record<string, McpStdioServerConfig>` with no field renames; the optional `type: 'stdio'` discriminator can be omitted.
- **`Skill` tool name is runtime-only.** The `Skill` token appears in the SDK runtime bundle (`sdk.mjs`) but is not in the static type definitions, so only a runtime spike (above) can confirm it is present in the live tool surface; the migration cannot rely on static-type compile-time narrowing for it.

## Out of Scope

- Fixture-facing permission knobs (`permissionMode`, `allowedTools`, `disallowedTools`, custom `canUseTool`). Fixtures continue to see today's behavior: every tool runs, ask-user reactions resolve through the bus.
- Fixture-facing arbitrary system-prompt replacement. The driver always uses the Claude Code preset for CLI-faithful skill evaluation; future work can add explicit prompt-variant evals if needed.
- Migrating stdio-based MCP mock servers to in-process SDK MCP servers. The existing forking model is preserved.
- Codex (exec mode) migration to an equivalent SDK-driven path. Codex (app-server) already has a real handshake; Codex (exec) remains subject to the existing fail-fast preflight.
- Cursor migration. Cursor is still in the `noninteractive` capability tier and continues to use today's mechanisms.
- Visual previews on ask-user options (`toolConfig.askUserQuestion.previewFormat`). Future enhancement.
- The `defer` hook decision for evals that exceed a single host process lifetime. Future enhancement for batch runners.
- The mid-session permission-mode change API. Not needed for current eval flows.
- Streaming-input multi-turn pattern (a single long-lived `query()` with a user-message async iterable). The resume-per-turn model is simpler and matches the existing per-turn lifecycle.
- Subagent (`Task` tool) ask-user support. Not exposed by the SDK by design; documented as a limitation.
- Adopting the broader observability stream the SDK exposes (hook lifecycle, task lifecycle, partial assistant messages, files-persisted events, rate-limit events, prompt-suggestion messages). Worth its own follow-up PRD.
- A fixture-file API to set `pathToClaudeCodeExecutable` per trial. The override is exposed only through run-level `AgentOptions.claudeCodeExecutable` and process env `PATHGRADE_CLAUDE_CODE_EXECUTABLE`; fixture-authored per-trial binary selection is deferred until a use case appears.

## Further Notes

- This PRD tracks Issue #38 ("Non-interactive mode masks the ask-user handshake — can't verify answer consumption"). The migration directly closes the gap the issue describes.
- The migration is a targeted capability upgrade, not a claim that the SDK default environment is identical to a user's interactive CLI. The driver opts into the Claude Code prompt preset and controls filesystem setting sources so Pathgrade evaluates skills in a reproducible Claude Code-like environment.
- The SDK ships per-platform `claude` binaries as optional npm dependencies. Install size will increase by tens of megabytes; this should be called out in the install section of the user guide. Users who already have the CLI installed and want to avoid the duplicate footprint can skip the optional dependency at install time and set the override path.
- Worth filing alongside this PRD: an upstream feature request to Anthropic for explicit documentation of the `permissionMode: "default"` × `canUseTool` × `AskUserQuestion` interaction, so the verification spike under "Testing Decisions" can be retired once the docs are updated.
- Worth filing alongside this PRD: an upstream feature request for a documented "CLI-equivalent but hermetic" SDK recipe that combines the Claude Code prompt preset with controlled setting sources and disabled ambient memory.
- The same architectural pattern (SDK-driven driver, custom spawn, `canUseTool` for ask-user routing) is the obvious next step for Cursor once that integration is ready to migrate. This PRD does not pre-commit to the Cursor path but leaves the door open.
