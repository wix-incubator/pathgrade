Source PRD: `docs/prds/2026-05-05-claude-sdk-agent-driver.md`

## Merge model

All nine slices land on the `claude-sdk-agent-driver` branch (this branch — same one that holds the PRD and these issue drafts). `master` receives a single squashed PR after #009 passes verification. The PRD requires "no transport flag, no half-state" on `master`, so individual slices are not master-mergeable on their own — they accumulate on this branch and the squashed PR is the atomic unit visible to consumers. PRD docs and issue drafts ride along in the same squashed PR.

## Completion protocol

When landing a slice, before adding the Progress Log row:

1. Open the slice's issue file and walk every `- [ ]` in **Acceptance criteria**.
2. For each item, verify it against the merged diff and the new/updated tests — not against memory, not against the PRD, not against this tracker. The issue file is the contract.
3. Flip `- [ ]` → `- [x]` only when verified. If the landed implementation deviates from the wording, annotate the checked line inline (`*(Deviation: ...)*`) and pin the rationale in **Completion Notes** below — do not silently rewrite the criterion.
4. Then update **Overall Status**, the **Issue Drafts** table status column, and append a Progress Log row.

**Do not invoke any `ck-*` skill while working on this migration.** The Creator Kit skill family (`ck-new`, `ck-status`, `ck-utility-pr`, `ck-utility-code-review`, `ck-flow-*`, `ck-product-*`, `ck-ux-*`, `ck-research-*`, etc.) is scoped to a different product and will pull in unrelated workflow assumptions. Use the standard tools — the issue files, the PRD, and direct code/test edits — to land each slice.

- **#006 land notes (2026-05-06).**
  - **Bridge surface widened, not replaced.** `createAskUserBridge` still returns a `CanUseTool` callable so #004's existing call sites and tests (`const canUseTool = createAskUserBridge(...); canUseTool(...)`) keep working without touch-ups; the new `lastError()` and `clearLastError()` are attached as function properties on the same callable. The driver now references it as `bridge` (with `bridge.lastError()` re-thrown post-stream), but the SDK still receives a plain `CanUseTool`.
  - **Decline detection rule.** `{ behavior: 'deny', message: 'User declined to answer' }` fires only when `resolution.answers.length > 0 && every answer.source === 'declined'`. Mixed-disposition resolutions (one declined among reactions/fallbacks) still allow with whatever non-empty answers the bus produced — at least one question was actually answered. The empty-array guard is defensive: a zero-question batch (impossible per SDK constraints, but cheap to guard) keeps the existing allow path rather than silently denying.
  - **Bus rejection no-retry rule (in scope for #006).** A bus rejection (timeout or otherwise) does not recover on retry — a missing/slow subscriber is a configuration problem, not a transient agent error. Without the `instanceof AskBusTimeoutError` short-circuit in `runTurn`'s retry loop, the runner spent ~6s burning two `RETRY_DELAY_MS * (attempt+1)` waits before reporting the same `'error'` completion. Issue criterion 2 ("causes the conversation runner to report an error completion") implies prompt failure; the short-circuit is the smallest change that satisfies that intent. Mirrors the existing `AgentCrashError` precedent.
  - **`questionTexts` populated on both `'error'` and `'first-option'` paths.** When `'first-option'` degrades to error (no options / isSecret), the unmatched signal still carries the full batch's question texts, not just the degraded one. The completion detail uses the *first* text only (bounded length); structured consumers see the full list. The `'decline'` disposition does NOT emit an unmatched signal (existing semantics preserved) — the deny shape on the SDK side is what surfaces decline.
  - **`pickAnswerSource` heuristic carried forward unchanged.** #004 noted that mixed-disposition batches tag the entire ToolEvent with the most-specific source (`reaction` > `fallback` > `declined`). #006 did not need to revisit this — the same heuristic now also drives whether the SDK sees `allow` or `deny` (only an all-declined resolution turns into a deny). If a future per-question source breakdown is needed on the envelope, that is its own slice; both #005 and #006's tests exercise the single-tag shape today.
  - **What `'declined'` on the answer store means for the projector.** When the bridge denies, it still records `{ answers: <empty-string map>, source: 'declined' }` on the per-turn answer store before returning. The projector merges that onto the `AskUserQuestion` ToolEvent envelope just like the happy path — so scorers see a uniform `{ ...AskUserQuestionInput, answers, answerSource: 'declined' }` shape on declined turns, instead of the `'unknown'` boundary stamp #002 left on un-merged blocks. This was implicit in the issue's "tests cover declined answers" criterion; the explicit assertion lives in `tests/claude-ask-user-bridge.test.ts`.

- **#008 land notes (2026-05-06).**
  - **Why two unions for assistant-message source.** Criterion 1 ("`VisibleAssistantMessageSource` value `blocked_prompt` are removed from active source types and agent turn results") and criterion 5 ("snapshot/log readers keep read-only parsing of those fields so historical snapshots predating the migration still load") pull in opposite directions when they share the same union: a single `VisibleAssistantMessageSource = 'assistant_message' | 'blocked_prompt'` either keeps the legacy literal (failing 1) or drops it (breaking the legacy snapshot test fixtures at `tests/run-snapshot.test.ts:75,80,129` that cast `{ assistant_message_source: 'blocked_prompt', ... } satisfies LogEntry`). Net effect: no writer can produce `'blocked_prompt'` (the active type's union doesn't include it; a hypothetical writer that constructed a turn-result with that literal would fail tsc), but the loader can read it from disk and the legacy fixtures still typecheck.
  - **Why `buildSyntheticAgentResultLogEntry` was deleted.** Not in the issue's named-deletions list, but it's the only emitter that ever set `synthetic_blocked_prompt: true` and `blocked_prompt_source_turn`, and its only call sites were inside the now-deleted blocked-prompt-queue replay branches in `converse.ts` (synthetic agent message during queue replay) and `chat.ts` (chat replay). Deleting it removes that footgun.

With the writer gone, the filter never matches on fresh runs, but it keeps legacy-snapshot-derived diagnostics shape-stable. Aligned with criterion 5's "readers keep read-only parsing" rule. Removing it would technically be safe (only fresh snapshots get rendered in practice) but fragile against any historical-snapshot inspection workflow, and the filter is a one-line clause with no maintenance cost.
